# OFD编辑器 PRD 骨架（面向政务/企业）

## 1. 文档信息

- 产品名称：OFD 编辑器（暂定）
- 文档类型：PRD 骨架 v1.0
- 目标版本：MVP（P0）/ 专业版（P1）/ 平台版（P2）
- 适用范围：政务公文、企业合规文档、电子档案场景

## 2. 背景与目标

### 2.1 背景
- 市场已有 OFD 产品多数聚焦“阅读”或“签章”单点能力，编辑与流程协同能力碎片化。
- 政务/国企对 OFD 的诉求已从“可打开”升级为“可生产、可归档、可审计、可集成”。
- 企业端需要可控、私有化、可二开的文档中台能力。

### 2.2 产品目标
- 打造一款符合 OFD 标准、支持编辑审阅签章归档的一体化产品。
- 形成“客户端高效编辑 + 服务端集中治理”的双端架构。
- 支持与 OA、档案、签章、发票系统的标准化集成。

### 2.3 非目标
- 不在 P0 阶段覆盖复杂排版设计工具（如专业 DTP 全能力）。
- 不在 P0 阶段实现跨语言 AI 内容生成能力。

## 3. 用户与场景

## 3.1 目标用户
- 公文员：编辑红头文、套版、校对、导出归档。
- 法务/合规岗：审阅修订、签章验章、留痕审计。
- 档案管理员：批量归档、元数据校验、检索复核。
- IT 管理员：权限策略、系统集成、稳定运行保障。

## 3.2 核心场景
- 场景 A：公文起草 -> 多人审阅 -> 加盖电子章 -> 归档。
- 场景 B：历史 PDF/Office 文档批量转换 OFD 并归档。
- 场景 C：跨系统在线预览、验章、审批流流转。

## 4. 需求范围（按版本）

## 4.1 P0（0-3 个月）MVP
- OFD 阅读与基础编辑（文本、图片、页面对象）。
- 批注与基础修订（高亮、删除线、文本注释）。
- 单章签章与基础验章。
- OFD/PDF 双向转换（优先准确率）。
- 本地打印与导出。

## 4.2 P1（4-7 个月）专业版
- 修订模式完善（插入、替换、移动、审阅意见回复）。
- 骑缝章、关键字定位章、批量盖章。
- 模板中心（公文模板、页眉页脚、编号规则）。
- 安全策略（水印、权限控制、受控阅读）。
- 批量任务中心（转换、验章、重试、告警）。

## 4.3 P2（8-12 个月）平台版
- 服务化 API 与 SDK（Java/C++/.NET）。
- 集中审计与全链路日志。
- 归档连接器（OA/档案系统标准接口）。
- 多租户策略、组织级权限与运营看板。

## 5. 功能清单（MVP 级）

## 5.1 文档编辑
- 新建/打开/保存 OFD。
- 页面级操作：插入页、删除页、旋转页、页面顺序调整。
- 元素级操作：文本修改、图片替换、对象对齐、撤销重做。

## 5.2 审阅协作
- 注释类型：文本、图形、手写标注。
- 修订显示：按作者/时间筛选、接受/拒绝修订。
- 导出审阅报告（变更摘要）。

## 5.3 签章验章
- 签章：单章盖章、签章位置拖拽、时间戳扩展位。
- 验章：签章有效性、证书状态、文档完整性结果展示。
- 签章日志：操作者、时间、证书标识、结果状态。

## 5.4 转换打印
- OFD <-> PDF（首批），Office 作为 P1 扩展。
- 批量转换队列、失败重试、错误原因归类。
- 打印控制：页码范围、缩放、批注/签章可见性。

## 5.5 安全与审计
- 权限策略：复制、打印、导出控制。
- 水印策略：静态水印/动态水印。
- 审计日志：查看、检索、导出。

## 6. 关键业务流程

## 6.1 公文生产流程
- 起草 -> 编辑 -> 审阅 -> 修订确认 -> 电子签章 -> 验章 -> 归档。

## 6.2 批量归档流程
- 批量导入 -> 格式校验 -> 批量转换 -> 元数据补全 -> 验章 -> 入库。

## 7. 体验与交互原则

- 所见即所得：编辑视图与打印视图一致。
- 高可理解：签章/验章结果可视化且可追溯。
- 低学习成本：常用操作路径不超过 3 次点击。
- 兼容优先：异常文档给出明确修复建议而非静默失败。

## 8. 非功能需求

- 性能：100 页文档打开时间 <= 3 秒（主流办公机）。
- 稳定性：核心功能可用性 >= 99.9%（平台版）。
- 安全：全链路操作审计；敏感日志脱敏存储。
- 兼容：覆盖主流国产操作系统与办公终端环境。

## 9. 验收指标（建议）

- 转换准确率：政务公文样本 >= 98%。
- 验章成功率：标准样本 >= 99.5%。
- 批量任务成功率：稳定运行场景 >= 99%。
- 缺陷控制：P1 阶段严重缺陷（S1）清零后发布。

## 10. 风险与应对

- 风险 1：复杂版式转换失真。
  - 应对：建立高优先级样本库和规则回归机制。
- 风险 2：多厂商签章兼容性问题。
  - 应对：建立跨厂商证书与签章互验测试矩阵。
- 风险 3：项目后期集成成本失控。
  - 应对：前置 API 规范与连接器边界定义。

## 11. 里程碑

- M1：需求冻结 + 原型评审 + 技术选型（第 4 周）
- M2：P0 开发完成 + 联调测试（第 12 周）
- M3：P1 发布（第 28 周）
- M4：P2 发布（第 52 周）
